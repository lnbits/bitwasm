# WIP-0002: Receipt Format v0 and Canonical Hashing

* **Title:** Receipt Format v0 and Canonical Hashing
* **Author:** bitWASM Contributors
* **Status:** Draft
* **Type:** Consensus
* **Created:** 2026-03-04

---

# Abstract

This proposal defines the **bitWASM execution receipt format (v0)** and the **canonical hashing rules** used to identify and sign execution results.

The receipt is the fundamental object in the bitWASM protocol. It represents the result of executing a WASM contract with a given input and state.

Federation nodes produce and sign receipts after deterministic execution. The receipt hash becomes the canonical identifier for the execution result and is used for consensus, verification, and settlement.

---

# Motivation

bitWASM requires a deterministic and verifiable way for federation nodes to agree on contract execution results.

The receipt provides a structured representation of:

* the program executed
* the previous state
* the input data
* the resulting state
* gas usage
* settlement outputs

By standardizing the receipt format and hashing rules, all nodes can independently verify that execution results match and produce identical signatures.

---

# Specification

## Receipt Overview

A **receipt** represents the deterministic result of executing a WASM program.

Conceptually:

```
receipt = execute(
    wasm,
    state_root_before,
    input
)
```

Execution produces:

```
(state_root_after, outputs, gas_used)
```

These values are recorded in the receipt.

---

# Receipt Fields

The receipt structure is defined as follows.

| Field               | Type    | Description                                             |
| ------------------- | ------- | ------------------------------------------------------- |
| `bitwasm_version`   | uint16  | Protocol version                                        |
| `vm_id`             | string  | Execution environment identifier                        |
| `wasm_hash`         | bytes32 | SHA256 hash of the WASM bytecode                        |
| `state_root_before` | bytes32 | State root prior to execution                           |
| `input_hash`        | bytes32 | Hash of the execution input                             |
| `state_root_after`  | bytes32 | State root after execution                              |
| `outputs_hash`      | bytes32 | Hash of settlement outputs                              |
| `gas_used`          | uint64  | Gas consumed during execution                           |
| `fee_sats`          | uint64  | Execution fee in satoshis                               |
| `expiry_height`     | uint32  | Bitcoin block height after which the receipt is invalid |
| `federation_id`     | bytes32 | Identifier of the federation executing the contract     |

---

# Execution Environment Identifier

The `vm_id` field specifies the execution environment.

Example:

```
wasm32-bitwasm-v0
```

Future WIPs may define additional VM profiles.

---

# Input Hash

The `input_hash` field commits to the execution input.

Inputs may include:

* user parameters
* transaction metadata
* contract invocation details

Input encoding must be deterministic.

The hash is computed as:

```
input_hash = SHA256(input_bytes)
```

---

# Outputs Hash

The `outputs_hash` commits to the settlement outputs generated by execution.

Outputs may include:

* Bitcoin payment outputs
* updated container state commitments
* federation fee allocations

Outputs must be serialized deterministically before hashing.

```
outputs_hash = SHA256(outputs_bytes)
```

---

# Canonical Serialization

All receipts must be serialized in a canonical format before hashing.

Receipt serialization order:

```
bitwasm_version
vm_id
wasm_hash
state_root_before
input_hash
state_root_after
outputs_hash
gas_used
fee_sats
expiry_height
federation_id
```

All integers are encoded in **big-endian format**.

Strings are encoded as:

```
length || bytes
```

where length is a 16-bit unsigned integer.

Binary values are encoded directly as fixed-length byte arrays.

---

# Canonical Receipt Hash

The receipt hash is defined as:

```
receipt_hash = SHA256(serialized_receipt)
```

This hash uniquely identifies the execution result.

Federation nodes sign the `receipt_hash`.

---

# Federation Signatures

Federation nodes produce signatures over the canonical receipt hash.

Example:

```
signature = sign(node_private_key, receipt_hash)
```

Federations may aggregate signatures using threshold schemes.

Example schemes:

* Schnorr threshold signatures
* FROST
* MuSig2

Signature aggregation mechanisms are defined in future WIPs.

---

# Determinism Requirements

All federation nodes must produce identical receipts for the same execution.

To guarantee this:

* WASM execution must be deterministic
* serialization must be canonical
* hashing must follow this specification exactly

If nodes produce different receipt hashes, the execution result is considered invalid.

---

# Backwards Compatibility

This WIP defines **Receipt Format v0**.

Future protocol versions may introduce new fields or serialization formats.

Receipts must always include the `bitwasm_version` field so nodes can correctly interpret the format.

---

# Security Considerations

Incorrect serialization or hashing could lead to consensus failures.

Implementations must ensure:

* strict adherence to canonical encoding
* deterministic execution environments
* consistent hashing algorithms

Federation nodes must refuse to sign receipts that do not match the canonical specification.

---

# Reference Implementation

Reference implementations should include:

* receipt serialization library
* canonical hashing functions
* deterministic encoding utilities
* receipt validation logic

Test vectors should be provided to ensure consistent behavior across implementations.

---

# Conclusion

The receipt format is the core data structure of the bitWASM protocol.

By defining a canonical receipt structure and hashing mechanism, this proposal enables deterministic agreement among federation nodes and provides a verifiable record of contract execution results.
